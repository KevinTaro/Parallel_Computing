# 效能瓶頸分析方法論 — Performance Bottleneck Playbook

> **目的**：給任何在其他專案做效能最佳化的 AI（或人）一套可複製的邏輯，
> 用來**先找對瓶頸、再動手**，避免重蹈本專案的覆轍 ——
> 花了 9 個版本去最佳化一個只佔 **1%** 執行時間的地方,結果做出**負最佳化**
> (GPU 版本比單核 CPU 還慢)。
>
> 本文分四段:**如何發現問題 → 如何分析問題 → 如何規劃解法 → 如何選出最佳解**,
> 每段都先講本專案真實發生了什麼(案例),再抽成通用規則(playbook)。

## 背景:這個專案是什麼、做了什麼 — Context for outside readers

> 這節寫給**沒看過這個專案**的人/AI。後面所有案例數字都出自這裡描述的實驗。
> 你不需要懂病理影像,只需要知道這是一個「固定任務 × 32 種平行化策略」的對照實驗。

**這是什麼專案**
一個平行計算的**方法論研究**,不是一個要出貨的產品。它挑了一個固定任務,
然後用 **32 個版本(v0–v32)** 分別實作不同的最佳化策略,系統性地量測
「哪種平行化有效、為什麼」。所有版本功能完全相同,差別只在「怎麼把工作分配給
CPU/GPU、以及瓶頸在哪」。

**固定任務(所有版本共用)**
輸入一張 gigapixel 的 WSI(whole-slide image,病理全切片影像,可達數萬×數萬像素)。
用滑動視窗切出候選 patch(1024×1024),對每個 patch:轉灰階 → 算白/黑像素比例 →
比例過高(純白背景或純黑空洞)就丟掉,留下的座標供後續訓練使用。
說穿了就是「**大量讀取小圖 + 對每張做很輕的像素統計**」。

**正確性約束(貫穿全程的鐵律)**
所有版本留下的座標集合必須與單核 CPU 基準**逐位元一致**(Jaccard = 1.0)。
灰階用整數公式 `(R·19595+G·38470+B·7471+32768)>>16` 精確複製 PIL 的 `convert('L')`。
「更快但答案不同」一律視為壞掉,不是更好。

**做了哪些版本(三條主線)**
- **線 A(v0–v10):CuPy 計算線。** 把像素統計從 CPU NumPy 搬到 GPU CuPy,
  逐步加批次、pinned memory、async streams、fp16、fused kernel、全部堆疊(v9)。
  → **這條線就是「最佳化了非瓶頸」的災難現場**,是本文的主要教材。
- **線 B(v11–v22):GPU 解碼線。** 發現真瓶頸是 CPU 解 JPEG 後,直接繞過
  OpenSlide,把 **JPEG 解碼本身搬上 GPU**(nvJPEG / 手寫 CUDA kernel),
  拆掉那個 98% 的地板。最佳版 v16 端到端 62×。
- **線 C(v23–v32):在 GPU 解碼管線上,把線 A 的每個概念重演一次**,
  交叉驗證結論。

**測試硬體(案例數字有兩台機器,別混用)**
- 小切片實驗:GTX 1060 3GB(Pascal,fp16 只有 fp32 的 1/64)+ 8 核 CPU。
  → 出處 `RESEARCH_RESULTS.md`。
- 大切片實驗:RTX 5090 32GB + ~20 核 CPU。→ 出處 `FINAL_REPORT.md`、`V1_V32_COMPARISON.md`。

**我(整理本文件的 AI)做了什麼**
我讀完這個專案的全部報告與程式碼,把它「一路走過的彎路與最後找到的正解」
反推成一套**通用的、可照抄的瓶頸分析流程**,寫成你正在看的這份文件。
本文件不含新的實驗,只把既有的真實量測數據提煉成方法論。核心來源:
`RESEARCH_RESULTS.md`(小切片端到端 + kernel 分解 + ablation)、
`FINAL_REPORT.md` 與 `V1_V32_COMPARISON.md`(大切片全版本對照)、
`hardware_probe.py`(roofline 兩端量測的探針邏輯)。

## 0. 一句話總結(TL;DR)

**先量測,再最佳化。** 在你知道時間花在哪裡之前,任何最佳化都是猜測;
而猜錯地方的最佳化,不只是白做,還可能是**負的**——你增加了複雜度、傳輸、
同步成本,卻沒碰到真正的瓶頸。

數學上這叫 **Amdahl 定律**:如果某部分只佔總時間的比例 `p`,
就算你把它加速到「無限快」,整體加速上限也只有 `1 / (1 - p)`。

> 本專案的 GPU kernel 佔 `p = 1.1%` → 理論最大整體加速 = `1/(1-0.011) ≈ 1.011×`。
> 換句話說:**把 GPU 計算最佳化到完美,也只能讓整體快 1.1%。** 我們卻在這 1.1%
> 上做了 v1–v9 九個版本、七層最佳化。真正的 98.9% 一直沒被碰到。

## 1. 如何「發現」問題 — How the problem was found

### 1.1 本專案發生了什麼

研究目標是「用 GPU (CuPy) 加速 WSI 病理切片的 patch 組織篩選」。
假設寫在計畫書裡:**v9 全最佳化堆疊,預期比多核 CPU 快 8–15×**。

我們老老實實做完了 v1–v9:逐 patch → 批次 → hybrid → pinned memory →
async streams → fp16 → fused kernel → 全部堆疊。工程完整度很高。

然後跑 end-to-end benchmark,結果是一記耳光:

| 版本 | 策略 | 時間 | vs 單核CPU (v0a) |
|---|---|---:|---:|
| v0a | 單核 CPU 基準 | 3.96s | 1.00× |
| **v0b** | **多核 CPU** | **1.32s** | **3.01×** ← 贏家 |
| v1 | 逐 patch GPU | 4.84s | **0.82×** ← 比單核還慢 |
| v2 | 批次 GPU | 5.38s | **0.74×** |
| v3 | hybrid | 5.47s | **0.72×** |
| v6 | fp16 GPU | 5.28s | **0.75×** |
| **v9** | **全最佳化堆疊** | **4.73s** | **0.84×** ← 最強 GPU 版,仍比單核慢 |

**這就是「負最佳化」的鐵證:** 我們加上了 GPU、加上七層最佳化,
成品卻比一個 `for` 迴圈的單核 CPU 還慢。投入的工程越多,離目標越遠。

### 1.2 發現問題的關鍵動作:量測「分解時間」,不是量「總時間」

總時間只告訴你「慢」,不告訴你「哪裡慢」。真正戳破假象的,是在 GPU 工作
**兩側掛上 CUDA event 計時器**,把 kernel 時間從 wall-clock 裡切出來:

| 版本 | 總時間 | kernel 時間 | kernel 佔比 | 其餘(I/O + Python) |
|---|---:|---:|---:|---:|
| v9 ultimate | 4.732s | **0.052s** | **1.1%** | 4.680s |

**52 毫秒。** 這就是我們花了九個版本最佳化的東西的全部大小。
剩下 98.9% 是 `openslide.read_region` 在 CPU 上解 JPEG + Python 讀取迴圈。

### 1.3 → 通用規則:如何發現你的真瓶頸

1. **先建一個誠實的基準線,而且要有對照組。**
   本專案的救命設計是同時保留了 `v0a`(單核)和 `v0b`(多核)兩個 CPU 基準。
   沒有 v0a,你不會發現 GPU 版是「負」的——你會以為 5 秒就是這任務該有的樣子。
   **對照組是照妖鏡。**

2. **量測時間的「分解」,不是「總和」。**
   在每個可疑階段兩側放計時器(CUDA event / `time.perf_counter` / profiler),
   得到一張「每階段佔幾 %」的表。**你要找的不是最慢的絕對值,是最大的佔比。**

3. **對每一個「我以為是瓶頸的地方」都問:它佔總時間幾 %?**
   如果答案是個位數 %,停手——這裡不值得最佳化,不管它多誘人、多好玩、
   多符合你原本的假設。

4. **警訊清單(看到就代表你可能在錯的地方使力):**
   - 你的最佳化版本比「最笨的基準」還慢或差不多 → 你沒碰到瓶頸。
   - 你加了很多層最佳化但總時間幾乎不動 → 你在優化雜訊。
   - 你的加速比遠低於理論預期(預期 8×,實測 0.8×)→ 假設錯了,回去量測。
   - profiler 顯示你的「熱點函式」其實佔比很小 → 熱點在別處(常是 I/O、
     序列化、鎖、同步、記憶體配置)。

## 2. 如何「分析」問題 — How the problem was analyzed

### 2.1 本專案發生了什麼

拿到「kernel 只佔 1.1%」這個數字後,我們寫了 `hardware_probe.py`——
一支**不猜、只量**的探針,把 roofline 的兩端各自量出來:

```
GPU luma kernel 吞吐(資料已在 GPU 上)  : ~17,500 patch/s
OpenSlide 單執行緒解碼吞吐             : ~27 patch/s
                                        ─────────────────
比值                                    : GPU 被餵料餓死 ~640×
```

結論白紙黑字:**GPU 不是慢,是被餓死(starved)。** 管線以 27 patch/s 餵它,
它有能力吃 17,500 patch/s,於是它 98% 的時間在**閒等**。

同時 probe 還量了一件決定「解法方向」的事——**OpenSlide 解碼時會不會釋放 GIL**:

```
單執行緒     :  27 patch/s   (1.0×)
執行緒 ×8    :  93 patch/s   (3.4×)   ← 會 scale → 代表 read_region 解碼時釋放了 GIL
行程   ×8    : 相近          → 多執行緒就夠,不必付多行程的 IPC 成本
```

這一步把「該怎麼修」從猜測變成**資料驅動的必然結論**。

### 2.2 分析時用到的三個思考工具

1. **Amdahl 定律(先算天花板,再決定值不值得做)。**
   最佳化某部分前,先算「就算它變成 0 秒,整體能快多少」。
   `p=1.1%` → 上限 `1.011×`。這個數字一算出來,v1–v9 的整個方向就被判死刑了。
   **這一步只要一行算術,卻能省下九個版本的白工。**

2. **Roofline / 兩端量測(找出是誰在等誰)。**
   把管線拆成「供給端(I/O、解碼、傳輸)」和「消費端(compute)」,各自量吞吐。
   誰的吞吐低,誰就是瓶頸;高吞吐的那端在空等。
   本專案:供給 27 patch/s ≪ 消費 17,500 patch/s → 瓶頸 100% 在供給端。

3. **分離「單元成本」與「管線成本」。**
   本專案有個容易騙人的真相:**GPU 的計算優勢是真的**——
   單獨量 compute-only,1024px patch GPU 比 CPU 快 6.98×,2048px 快 8.37×。
   但這個真實的優勢在完整管線裡**被 I/O 完全蓋住**。
   教訓:**一個最佳化在「微觀基準」上是真的贏,不代表它在「端到端」會贏。**
   永遠要問:這個微觀優勢,佔端到端的幾 %?

### 2.3 → 通用規則:如何分析瓶頸

- **先算 Amdahl 天花板,再決定要不要動手。** 一行算術否決一個月的方向。
- **量測 roofline 兩端**,找出「誰餓死誰」。瓶頸永遠是低吞吐那端。
- **區分「這東西快不快」和「這東西是不是瓶頸」。** 兩者無關。
  快的東西可以不是瓶頸(GPU compute);慢的東西才是(CPU 解碼)。
- **不要相信直覺熱點,量出來的才算。** 直覺常指向「看起來計算量大」的地方
  (矩陣運算、kernel),但真瓶頸常在無聊的地方:I/O、序列化、鎖、
  記憶體配置、同步點、Python 直譯器開銷。

## 3. 如何「規劃」解法 — How solutions were planned

### 3.1 本專案發生了什麼

一旦確定瓶頸是「CPU 解碼 + 序列餵料」,解法空間就**被瓶頸本身定義了**,
不再是漫無目的地堆 GPU 技巧。我們產生了兩層解法,由淺入深:

**第一層:不改地板,先把地板並行化(攻擊供給端的並行度)**
- `v0b`:多行程 CPU 讀取 → 3.0× (最簡單、立刻可用的正解)
- `v10`:單行程內多執行緒讀取(GIL 在解碼時是 free 的)+ GPU 過濾 → 6.77×
- 核心洞見:**慢的部分(解碼)並行化,快的部分(過濾)留在 GPU 幾乎免費且被藏住。**

**第二層:直接拆掉地板(消滅瓶頸本身)**
- 瓶頸是「CPU 用 libjpeg 解 JPEG」→ 那就**別在 CPU 解**。
- `v11–v22`:自己解析 TIFF、讀原始壓縮 tile、把 **JPEG 解碼本身搬上 GPU**
  (nvJPEG 或手寫 CUDA kernel)。
- 這一步把原本佔 98% 的 CPU 解碼**整個移除**,瓶頸從 CPU I/O 變成 GPU 解碼吞吐。

**規劃時的關鍵:每個新版本都只針對「當下量出來的瓶頸」。**
- v19 共用一個 file object + lock → 量出「讀其實是序列的」→ v20 用 `os.pread` 真並行。
- v20 仍在主執行緒每批派發 → 量出「派發卡在 GPU 關鍵路徑」→ v21 用 producer thread 預取。
- v21 主執行緒還在序列做 destuff → 量出「destuff 成了新瓶頸」→ v22 把 destuff 也並行。

**這是正確的節奏:修一個瓶頸 → 重新量測 → 新瓶頸浮現 → 再修。**
每一步都由**新的量測**驅動,不是由「還有什麼技巧沒用上」驅動。

### 3.2 → 通用規則:如何規劃解法

1. **讓瓶頸定義解法空間,不要讓「你會的技巧」定義它。**
   問「怎麼消除/繞過/並行化這個特定瓶頸」,不要問「我還有什麼最佳化沒加」。
   v1–v9 是後者的災難:把「已知 GPU 技巧清單」跑完一遍,而清單跟瓶頸無關。

2. **解法優先序:先便宜的,再貴的。** 通常是:
   - (a) **並行化瓶頸**(多執行緒/多行程)——常常一步到位,如 v0b。
   - (b) **把瓶頸移出關鍵路徑**(預取、重疊、pipeline)——如 v10/v21。
   - (c) **消除瓶頸本身**(換演算法、換硬體路徑)——最強但最貴,如 v11 把解碼搬上 GPU。
   先試 (a),很多時候就夠了。本專案 v0b 一個 `Pool(cpu_count())` 就 3× 了。

3. **一次只打一個瓶頸,打完重新量。** 瓶頸會移動。修掉 A,B 就浮上來。
   不要一次堆五個最佳化——你會不知道哪個有效,而且很可能其中四個在打不存在的問題。

4. **保留一條「什麼都不做的最簡路徑」當對照。** v0b(純 CPU 多行程)一直是
   本專案的實用最佳解之一,提醒我們:**最好的解法有時不是最炫的那個。**

## 4. 如何「選出最佳解」 — How the best solution was chosen

### 4.1 本專案發生了什麼

最終在兩個維度上分別選出最佳解,而且**都是量出來的,不是宣稱的**:

- **不換硬體路徑時**:`v10`(並行 I/O + GPU)是「硬體允許的最快設計」,6.77×。
- **允許拆地板時**:`v16`(手寫 optimized CUDA 解碼 × 單執行緒餵料)
  大切片端到端 **62.0×**,解碼階段 **47×**,甚至**贏過 nvJPEG 固定功能硬體的 29×**。

值得注意的反直覺結果(全部靠量測才發現,靠猜一定猜錯):
- **async streams 幾乎沒用**:當瓶頸是 compute 本身時,傳輸重疊沒東西可藏,
  v18–v22 在 v16 之上幾乎零提升。
- **fp16 反而更慢**:在 Pascal 卡上 fp16 吞吐是 fp32 的 1/64,是**負最佳化**。
- **「終極堆疊」不是最快**:v18–v22 疊了最多層,卻輸給更簡單的 v16——
  因為它們最佳化的傳輸層早就不在關鍵路徑上了。**又一次「優化了非瓶頸」。**

### 4.2 選擇時的鐵律:正確性是一票否決

整個研究有一條貫穿始終的約束:**所有版本的保留座標集合必須與 v0a 逐位元一致
(Jaccard = 1.0000)。** 一個「更快但答案不同」的版本不是更好的解,是壞掉的解。

- 整數 luma 路徑刻意複製 PIL 的 `(R·19595+G·38470+B·7471+32768)>>16`,位元精確。
- fp16 / nvJPEG / 手寫 float-IDCT 會有 ±1 LSB 漂移,門檻邊緣 patch 可能翻轉決策——
  每個 benchmark 都**明確報出 kept-set 差異**,把「速度」和「正確性代價」擺在一起看。

### 4.3 → 通用規則:如何選最佳解

1. **用端到端 wall-clock 選,不用微觀基準選。** 微觀贏(GPU compute 8×)可以
   端到端輸(被 I/O 蓋住)。使用者感受到的是 wall-clock。

2. **「最多最佳化」≠「最快」。** 每一層最佳化都要能在**端到端**證明自己的貢獻
   (做 ablation:拿掉這層,總時間變化多少?)。不能證明的層,是複雜度負債,砍掉。
   本專案 ablation 顯示:async 加了 2× 記憶體、端到端零收益;fp16 是負的。

3. **正確性是一票否決,不是可談判項。** 先確立位元級/容忍度的正確性 gate,
   任何「更快」的版本必須先過 gate 才有資格比速度。並且**把精度代價量化報出來**。

4. **最簡單的夠用解,常常就是最佳解。** v0b 一個多行程 Pool 拿到 3×,
   零 GPU、零 CUDA、零 pinned memory。除非有明確的、量測支持的理由要更複雜,
   否則選簡單的。

## 5. 反模式速查表 — Anti-pattern checklist

在你動手最佳化**之前**,逐條自問。任何一條答「是」就先停下來量測。

| # | 反模式 | 本專案的化身 | 解藥 |
|---|---|---|---|
| 1 | **沒量測就最佳化**(assumption-driven) | 計畫書假設 GPU 快 8–15×,直接開做 v1–v9 | 先 profile,拿到每階段佔比表 |
| 2 | **最佳化非瓶頸**(佔比 <10% 的地方) | 猛攻佔 1.1% 的 GPU kernel | 先算 Amdahl 天花板,個位數 % 就別碰 |
| 3 | **沒有對照組/基準線** | (幸好有 v0a/v0b,才照出負最佳化) | 永遠留一個「最笨基準」當照妖鏡 |
| 4 | **信直覺熱點,不信量測** | 以為「計算」是瓶頸,其實是 I/O | profiler 說了算 |
| 5 | **把微觀勝利當端到端勝利** | GPU compute 真的快 8×,但被 I/O 蓋住 | 只用 wall-clock 端到端判定 |
| 6 | **堆技巧當作前進** | v18–v22 疊最多層卻輸給簡單的 v16 | 每層都要 ablation 證明端到端貢獻 |
| 7 | **一次改很多、無法歸因** | (本專案靠一次一版避開了) | 一次一個瓶頸,改完重量 |
| 8 | **忽略硬體/平台特性** | Pascal 上 fp16 是 1/64 → 負最佳化 | 量硬體真實能力(如 hardware_probe) |
| 9 | **為了速度犧牲正確性而不自知** | (靠 Jaccard=1.0 gate 擋住) | 正確性 gate 一票否決 + 量化精度代價 |
| 10 | **「假並行」當「真並行」** | v19 共用 fd+lock,讀其實是序列的 | 量並行後的實際吞吐,別信程式碼長相 |

## 6. 標準作業流程(可直接照抄的 checklist)

```
┌─ 發現 ────────────────────────────────────────────────┐
│ [ ] 建立誠實基準線 + 至少一個對照組(最笨的版本)        │
│ [ ] 端到端量總時間,確認「確實有問題」                    │
│ [ ] 分解時間:每個階段兩側掛計時器,得到「% 佔比表」       │
│ [ ] 找出最大佔比階段 = 候選瓶頸(不是最慢絕對值,是最大%)  │
└───────────────────────────────────────────────────────┘
┌─ 分析 ────────────────────────────────────────────────┐
│ [ ] 對候選瓶頸算 Amdahl 天花板 1/(1-p),值不值得做?      │
│ [ ] roofline 兩端量吞吐,確認「誰餓死誰」                 │
│ [ ] 確認瓶頸性質:I/O? 序列化? 鎖? 同步? 配置? 頻寬?    │
│ [ ] 量硬體/平台真實能力,排除平台特性造成的假設錯誤        │
└───────────────────────────────────────────────────────┘
┌─ 規劃 ────────────────────────────────────────────────┐
│ [ ] 讓瓶頸定義解法,不是讓「你會的技巧」定義              │
│ [ ] 依序考慮:(a)並行化 (b)移出關鍵路徑 (c)消除瓶頸本身    │
│ [ ] 先試最便宜的解(常常一步到位)                        │
└───────────────────────────────────────────────────────┘
┌─ 選擇 ────────────────────────────────────────────────┐
│ [ ] 一次只上一個最佳化,改完重新量端到端                  │
│ [ ] 每層做 ablation:拿掉它,端到端變化多少?零貢獻就砍   │
│ [ ] 正確性 gate 一票否決;量化任何精度代價                │
│ [ ] 用端到端 wall-clock 選最佳,不用微觀基準;偏好簡單解   │
│ [ ] 瓶頸移動了?回到「發現」重跑整個流程                  │
└───────────────────────────────────────────────────────┘
```

## 7. 給其他專案的 AI:三條最重要的話

1. **一行 Amdahl 算術,勝過一個月的最佳化直覺。**
   動手前先問「這部分佔幾 %、就算歸零整體能快多少」。個位數 % 就走開。

2. **對照組是照妖鏡。** 永遠保留一個「最笨、最簡單」的基準版本。
   沒有它,你永遠不知道自己是不是在做**負最佳化**——讓系統變得更慢還沾沾自喜。

3. **快 ≠ 是瓶頸;慢 ≠ 值得先修。** 瓶頸是量出來的,不是猜出來的,
   而且它會移動。每修一個就重新量。真正的瓶頸通常在無聊的地方
   (I/O、鎖、序列化、記憶體配置),不在你想炫技的地方。
