CPU to GPU Study

任務與核心基準

The Core Task & Evaluation Framework

專案概述

這是一個探討**平行運算與硬體加速**的研究專案。任務目標極其固定：

使用滑動視窗 (Sliding Window) 掃描醫療全玻片影像 (Gigapixel WSI)，並僅保留含有組織的切片 (Patches)。

  • 資料集： Philips Her2 WSI `.tiff`。
  • 不變法則： 運算邏輯在所有 30+ 版本中皆完全一致。
  • 正確性閘門： 每個版本必須完美重現 `v0a` 的過濾結果。

核心演算法 (Luma & Threshold)

1. 精確整數亮度轉換

為了確保 CPU 與 GPU 的位元級 (bit-for-bit) 一致性，採用 PIL 的精確整數 ITU-R 601 公式，避免浮點數誤差：

L=(R×19595+G×38470+B×7471+32768)≫16

2. 過濾條件

計算每張 Patch 中極端像素的數量：

  • > 白像素： 灰階值 > 230
  • > 黑像素： 灰階值 < 25

若任一極端像素比例 ≥ 0.9 (90%)，即視為背景或空洞並丟棄。

三條演進主線 (The Three Storylines)

這 30 餘個版本並非隨機嘗試，而是由三條清晰的技術脈絡所組成：

線 A: CuPy 計算線
(v1 - v10)

嘗試把「過濾計算」搬上 GPU。過程中逐步發現，真正的效能瓶頸其實是 CPU 的 JPEG 解碼。

線 B: GPU 解碼線
(v11 - v22)

拔掉 CPU 解碼的地板。將 JPEG 解碼本身也搬上 GPU (nvJPEG / CUDA)，並優化資料餵養管線。

線 C: 概念重演線
(v23 - v32)

在 GPU 解碼的全新世界中，重新實作一次 v1-v10 的概念，驗證底層優化原理的一致性。

線 A: CuPy 計算線 (v1 - v10)

把過濾計算移至 GPU，與尋找真瓶頸的旅程

v1: Naive (逐一切片 GPU化)

data_loader_v1_cupy_full.py v1_full

設計：最天真的策略。每張 Patch 各自進行一次 Host-to-Device 傳輸，並呼叫 GPU Kernel。

結果：這是一個**反例**。傳輸與 Kernel Launch 的開銷被放大數千倍，導致執行速度極差，甚至不如 CPU。

程式碼特徵：在 _keep_patch 中直接使用 cp.asarray 傳輸單張影像，並用 float(cp.sum) 強制同步拉回結果。

v2: Batch (整批處理)

data_loader_v2_cupy_batch.py v2_batch

設計：將 N 張 Patches 打包成 (N, H, W, 4) 陣列，進行**單次**傳輸，並沿空間軸 (axis=1,2) 向量化計算。

結果：大幅攤平了 Launch 開銷，GPU 開始發揮正向效益。證明了：關鍵不在用不用 GPU，而在有沒有批次餵料。

限制：高 VRAM 消耗。uint32 灰階中間值會吃掉大量記憶體。

v3: Hybrid (聰明路由)

data_loader_v3_cupy_hybrid.py v3_hybrid

設計：動態決策。當剩餘/尾數的 Patch 數量小於 gpu_threshold (預設 16) 時，退回使用 CPU NumPy 計算。

結果：避免在零碎區塊浪費 GPU 初始化資源，是最貼近真實部署彈性的架構。

正確性保證：CPU 與 GPU 路徑共用完全相同的 ITU-R 601 整數權重，結果與走哪條路徑無關。

v4: Pinned Memory (頁鎖定記憶體)

data_loader_v4_cupy_pinned_memory.py v4_pinned

設計：捨棄可分頁 (Pageable) 記憶體，改用 **Pinned Staging Buffer** 與預先配置的 Device Buffer。

  • 透過 PinnedMemoryPool 分配。
  • 允許 GPU DMA (Direct Memory Access) 直讀主機記憶體。
  • 免除了每次批次處理時配置記憶體 (malloc) 的抖動開銷。

意義：為後續的「非同步傳輸」打下基礎。

v5: Async Streams (非同步管線)

data_loader_v5_cupy_async.py v5_async

設計：實作真正的 雙緩衝 (Double-buffered) CUDA Streams。

運作原理： 分配兩個獨立的 Slot (含 Stream 與 Buffers)。當主機正在讀取「批次 k+1」時，GPU 正在另一條 Stream 上計算「批次 k」。

結果：將 I/O 傳輸時間成功隱藏 (Overlap) 在計算時間之後。這是純 CuPy 計算線在排程上的最高境界。

v6: Mixed Precision (精度實驗)

data_loader_v6_cupy_mixed_precision.py v6_mixed

設計：將灰階計算公式改用 float16 (fp16)，試圖節省一半的記憶體頻寬。

發現 1 (精度取捨)：fp16 只有 11-bit 尾數，會導致處於門檻邊緣的 Patch 決策翻轉，無法保證 100% 吻合 baseline。

發現 2 (硬體陷阱)：在 Pascal 架構 (如 GTX 1060) 上，fp16 的吞吐量被閹割，實際上跑得比 v2 還慢。只有 Volta 以後的架構才划算。

v7: Memory-Optimized (低 VRAM 救星)

data_loader_v7_cupy_memory_optimized.py v7_memopt

設計：針對 3GB VRAM 等低階顯卡設計。撰寫了 Fused (融合) uint8 Kernel。

原理：在單一 Kernel 中直接完成 `RGB -> 整數 Luma` 的計算與寫出。原本 v2 需要物化龐大的 `uint32` 中間陣列，v7 讓這個陣列「從不存在」。

結果：峰值 VRAM 佔用從輸入的 ~5x 銳減至 ~1.33x，徹底解決 OOM 問題。

v8: Optimized for 4060 (釋放 GIL)

data_loader_v8_cupy_optimized_4060.py v8_4060

重大轉折：第一次將優化火力轉向「餵料側 (Feed side)」。

設計：利用 ThreadPoolExecutor 建立多個 OpenSlide reader。
因為 libopenslide 在底層解碼時會釋放 Python 的 GIL，這實現了真正的多執行緒並行讀取。

結果：效能大幅躍升。這暗示了真正的瓶頸其實一直在 CPU 端。

v9: Ultimate GPU (集大成與消融)

data_loader_v9_ultimate_gpu.py v9_ultimate

設計：將 v1 到 v7 的所有運算優化（Pinned, Async, Fused Kernel 等 7 個 Layer）全部堆疊在一起，並採用設定驅動 (Config-driven)。

目的：建立一個可用於「消融研究 (Ablation)」的載體，可單獨開關每一層來量測貢獻。

反思：雖然它是 CuPy 計算線的最強體，但它優化的是「從來就不是瓶頸」的部分。

轉折點：Hardware Probe 診斷

hardware_probe.py Diagnostic

這個腳本是整個專案的靈魂。它量測了 PCIe 帶寬與 GPU 實際 Kernel 吞吐量。

殘酷的真相：數據顯示，GPU Luma 過濾可達 ~17,500 patches/s，但 CPU 單核讀取只有 ~27 patches/s。

結論：GPU 處於 98% 閒置狀態（被餓死），真正的瓶頸是 CPU 解碼！

v10: Parallel I/O (資料驅動的正解)

data_loader_v10_parallel_io_gpu.py v10_par_io

行動：依據 Probe 的結果，徹底放生對 GPU 核心的微調。

設計：建立一個 Reader-Thread Pool，每個執行緒擁有自己的 OpenSlide Handle，瘋狂並行解碼填入 Pinned Buffer，再批次丟給 GPU。

結果：這是硬體 (CPU 解碼) 允許下的最快設計。促使研究正式邁入下一階段：如何消滅 CPU 解碼。

線 B: GPU 解碼線 (v11 - v22)

拔掉 CPU 解碼地板，挑戰硬體極限

v11: First GPU-Decode (nvJPEG)

data_loader_v11_gpu_decode_5090.py v11_gpudec

革命性架構：繞過 OpenSlide。

自行解析 TIFF 標籤 (Tag 324/325)，直接讀取原始壓縮的 baseline JPEG 位元組。

交由 torchvision.io.decode_jpeg(device='cuda') 驅動 NVIDIA nvJPEG 硬體解碼器進行 GPU 批次解碼。

代價：nvJPEG 的 YCbCr->RGB 轉換與 libjpeg 有微小差異 (±1 LSB)，導致極少數 Patch 決策翻轉，打破了 100% 吻合的承諾。

v12 & v13: nvJPEG 餵料消融

v12_dec_mono data_loader_v12_gpu_decode_mono.py

Ablation Cell: 單執行緒讀原始 Tile + nvJPEG 解碼。

用於隔離「純特化 GPU 解碼」所帶來的效益 (相對 v0a)。
吞吐量：2378.2 p/s

v13_dec_multi data_loader_v13_gpu_decode_multi.py

Ablation Cell: 多執行緒讀原始 Tile + nvJPEG 解碼。

對比 v12 可精確量測「讀取並行化」的純粹加成。
吞吐量：1608.2 p/s (受限於 I/O 鎖定機制)

v14 & v15: Naive CUDA 解碼

v14_cmp_mono data_loader_v14_gpu_compute_mono.py

不依賴 nvJPEG，改呼叫手寫 CUDA 核心 (Naive 版)。

一個 CUDA Thread 處理一個 Tile。展示通用 CUDA 核心的解碼潛力。
吞吐量：3194.4 p/s

v15_cmp_multi data_loader_v15_gpu_compute_multi.py

多執行緒讀取 + Naive CUDA 解碼。

證明了不依賴第三方影像庫，直接在通用 GPU 核心上解碼醫療影像瓦片是極具競爭力的方案。

v16 & v17: Optimized CUDA 解碼

v16_opt_mono data_loader_v16_cuda_opt_mono.py

全專案吞吐量冠軍：4734.4 patches/sec (62.00x)

換上了 _optimized 版本的 CUDA 核心。利用暫存器緩衝、常數記憶體，以及 Fused YCbCr->Luma Kernel (不產生 RGB)，將硬體效能榨乾。

v17_opt_multi data_loader_v17_cuda_opt_multi.py

v16 + Threaded reads (多執行緒讀取)。

v18 & v19: Ultimate Pipeline (假並行現形)

v18_ult_mono ..._v18_cuda_ultimate_mono.py

終極管線： v16 kernel + Pinned Memory + 雙緩衝 Async Streams + Buffer Pool。

透過 submit() / fetch() 驅動，吞吐量穩定在 3247.2 p/s。

v19_ult_multi ..._v19_cuda_ultimate_multi.py

失敗的並行： 導入 Thread Pool 讀取。但所有 Thread 共用同一個 file object 與 lock，導致讀取實際上是序列化的 (Serial)。

效能反而跌至 2560.7 p/s。

v20: True Parallel Reads (真並行讀取)

data_loader_v20_cuda_ultimate_pread.py v20_ult_pread

修正 v19 的錯誤： 捨棄標準的 seek + read。

改用作業系統層級的 os.pread(fd, length, offset)。

這是一個原子性的系統呼叫 (Syscall)，不需鎖定檔案指標。徹底解放了 GIL，實現真正的多執行緒磁碟 I/O。

v21: Prefetch Pipeline (移出關鍵路徑)

data_loader_v21_cuda_ultimate_pipeline.py v21_ult_pipe

架構改造 (Producer/Consumer)：
v20 的讀取雖是平行的，但仍卡在主執行緒的 pool.map 派發上，導致 GPU 批次間出現空窗期。

v21 建立專屬的 Producer Thread 不斷預讀 (Prefetch) 批次資料放入 Queue，主執行緒 (Consumer) 只負責與 GPU 溝通 (submit / fetch)。讀取完全移出關鍵路徑。

v22: Parallel Read + Destuff (單 GPU 極限)

data_loader_v22_cuda_parallel_destuff.py v22_par_destuff

最後的阻礙： v21 的主執行緒仍需做 JPEG Byte-destuffing (將 \xff\x00 轉為 \xff)，這是一個沉重的 Python 序列化負擔 (每批次處理 ~190MB)。

終極解法： 將 Destuffing 任務也丟進 Reader Threads 處理。

至此，主迴圈的所有序列 Host 工作完全消失，執行速度純粹被 GPU 解碼吞吐量綑綁。這是單一 GPU 的理論極限架構。

CUDA 引擎深度解析

The Hand-Written CUDA JPEG Decoders (Naive vs Optimized vs Ultimate)

手寫 CUDA 核心系列

這三個版本都不使用 nvJPEG，完全依賴自研 CuPy RawKernel。三者輸出逐位元相同 (bit-identical)，差別純粹在效能。

gpu_jpeg_decoder.py Naive Baseline

一個 Thread 解碼一個 Tile，純靠「跨 tile 平行」。刻意不優化，使用全域記憶體與浮點數 IDCT。

gpu_jpeg_decoder_optimized.py Optimized

把核心調到榨乾顯卡：暫存器位元緩衝、8-bit Huffman LUT、常數記憶體、DC-only 快路徑、以及解碼與亮度計數融合。

gpu_jpeg_decoder_ultimate.py Ultimate

在 Optimized 之上再疊加**傳輸/重疊層** (Pinned memory, 雙緩衝 Async stream, 緩衝池)，驅動異步管線。

六大核心調校 (Why Optimized Wins)

將 Naive 的 1.55s 壓縮到 1.05s (突破 NVIDIA 硬體解碼器) 的六個平凡但致命的工程技巧：

1. 暫存器位元緩衝

(Register bit-buffer) 每讀 1 bit 就做 global load 太慢。改為 32-bit 暫存器累加器，一次讀 1 byte，記憶體流量銳減。

2. 8-bit Huffman LUT

絕大多數短碼 (≤8 bits) 直接透過常數記憶體「一次查表」解出，省去逐位元比對的迴圈。

3. 常數記憶體廣播

將所有熱點表 (Huffman, Quant, IDCT Cos) 放進 __constant__，利用 Warp 廣播讀取，節省全域頻寬。

4. DC-only 快路徑

平滑背景通常沒有 AC 係數。直接算出常數填滿區塊，跳過 1024 次 MAC 乘加運算。

5. 融合計數 (不產生 RGB)

這點贏過 nvJPEG！將 YCbCr->Luma->黑白計數融合成一個 Pass。**整張 RGB 從不被寫出**，VRAM 需求極小化。

6. 啟動參數調校

IDCT 區域變數極重，偏好低 Occupancy。實測發現 32 threads/block 效能最佳。

為什麼疊滿神級優化反而撞牆？

Ultimate 實作了所有傳輸與重疊層 (Pinned, Stream, Pool)，成功讓 GPU 等待 Host 的時間 (gpu-wait) 降到了完美的 **0.000s**。但總耗時卻幾乎不變。

  • > Compute-Bound 的詛咒： 在暖快取的 SSD 上，Host 讀取耗時極低 (~7%)，絕大部分時間都在 GPU 解碼。
  • > 藏起 7% 等於 0%： 管線成功把微小的 I/O 時間藏到 GPU 運算後面，但 GPU 依然要苦命算完數學。加上排程的微小開銷，甚至偶爾慢一點點。

結論：傳輸/重疊優化只有在 I/O 成為真實瓶頸 (例如冷快取、HDD 或 NFS) 時才有回報。

解碼器研究的關鍵副發現

  • 跨 Tile 平行是最大的第一槓桿 (1x → 13x)
    就算是最 Naive 的寫法，只要把近 2 萬張獨立的 Tile 丟到 GPU 同時解碼，就能將數十秒的牆鐘時間砍到個位數。
  • 核心調校是第二大槓桿 (13x → 47x / 62x)
    針對 GPU 架構特性（減少全域讀取、使用常數記憶體、減少 VRAM 寫出）進行的底層工程，是突破硬體極限的唯一途徑。
  • GPU 解碼後，多執行緒讀取反而有害 (Mono > Multi)
    實測發現 v17 < v16，v19 < v18。當解碼引擎上了 GPU，CPU 端單純的讀取已不再是瓶頸；此時開 20 條 Thread 去搶單一檔案的 Handle，只會徒增 GIL 與鎖定競爭。因此，最強的解碼核心都應該搭配 **Mono (單執行緒)** 餵食。

核心領悟 1 & 2

1. 批次化 > 用不用 GPU v1 → v2

單純將運算移交 GPU 不保證變快。傳輸與啟動開銷極高，只有將資料打包 (Batching) 攤平開銷，GPU 加速才具備經濟效益。

2. 優化要打在瓶頸上 v9 vs v10

v1-v9 拼命優化 GPU 計算，但診斷顯示 GPU 高達 98% 處於閒置狀態。
真正的瓶頸是 CPU 解碼！將解碼並行化 (v10) 才是治本的正解。

核心領悟 3 & 4

3. 拔掉地板 (Eliminate the Floor) v11 - v22

若 CPU 解碼速度永遠跟不上 GPU，就連根拔起，將 **JPEG 解碼本身** 移至 GPU 執行，徹底消滅 CPU 解碼帶來的硬性地板。

4. 「並行」不只是有沒有，而在哪跑 v19 → v20 → v21 → v22

- 假並行：共用 File Descriptor 與 Lock (v19)。
- 真並行：使用無鎖的 os.pread (v20)。
- 移出關鍵路徑：Producer/Consumer 架構 (v21)。
- 最終極限：連 Byte-destuff 都並行處理 (v22)。

核心領悟 5 & 6

5. 單 GPU 的物理極限 v22

當 Host 端餵料全並行，且所有序列工作搬離關鍵路徑後，耗時完全等於「單一 GPU 硬體解碼時間」。要再突破只能依賴多 GPU 分片。

6. 正確性貫穿全程 Validation

效能無意義，若不具備正確性。整數 Luma 路徑必須位元等同 PIL。使用 fp16 或是手寫 float-IDCT 所產生的 ±1 LSB 漂移，都必須被量化記錄 (Kept-set delta)。
